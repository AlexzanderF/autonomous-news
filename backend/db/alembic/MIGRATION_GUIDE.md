# Database Migration Guide

This guide explains how to create and apply database migrations using Alembic in this project.

## Prerequisites

- Ensure your virtual environment is activated
- Database must be running and accessible
- Environment variables in `.env` must be correctly configured

## Common Migration Workflows

### 1. Create a New Migration

After modifying `models.py`, navigate to the `db` directory and auto-generate a migration:

```bash
cd backend/db
alembic revision --autogenerate -m "descriptive_migration_name"
```

**Example:**
```bash
alembic revision --autogenerate -m "add_user_table"
```

This will:
- Compare your models with the current database schema
- Generate a new migration file in `alembic/versions/`
- Automatically detect added/removed tables, columns, indexes, and constraints

### 2. Review the Generated Migration

Always review the generated migration file in `alembic/versions/` before applying it:

- Check the `upgrade()` function contains the correct changes
- Verify the `downgrade()` function can properly rollback
- Ensure the `down_revision` points to the correct parent migration

### 3. Apply Migrations

Apply all pending migrations to bring the database to the latest version:

```bash
alembic upgrade head
```

### 4. Check Migration Status

View the current migration version:

```bash
alembic current
```

View migration history:

```bash
alembic history
```

View detailed history with verbose output:

```bash
alembic history --verbose
```

## Rollback Migrations

### Rollback One Migration

```bash
alembic downgrade -1
```

### Rollback to a Specific Revision

```bash
alembic downgrade <revision_id>
```

**Example:**
```bash
alembic downgrade a1b2c3d4e5f6
```

### Rollback All Migrations

```bash
alembic downgrade base
```

## Manual Migration Creation

If you need to create a migration manually (without autogenerate):

```bash
alembic revision -m "manual_migration_name"
```

Then edit the generated file and add your custom SQL or SQLAlchemy operations.

## Best Practices

1. **Always review autogenerated migrations** - Alembic may not detect all changes correctly
2. **Test migrations on a development database first** - Never run untested migrations on production
3. **Use descriptive migration names** - Makes it easier to understand migration history
4. **Keep migrations small and focused** - One logical change per migration
5. **Never edit applied migrations** - Create a new migration to fix issues
6. **Commit migrations to version control** - Migrations are part of your codebase

## Troubleshooting

### Migration Conflicts

If you get a "multiple heads" error, you need to merge the branches:

```bash
alembic merge -m "merge_branches" <revision1> <revision2>
```

### Database Out of Sync

If your database schema doesn't match your models and Alembic can't detect changes:

1. Check `alembic current` to see the current revision
2. Manually inspect the database schema
3. Create a manual migration if needed

### Reset Migrations (Development Only)

**⚠️ WARNING: This will delete all data!**

```bash
# Drop all tables
alembic downgrade base

# Reapply all migrations
alembic upgrade head
```
